# GrooveScribe V1 產品完整度標準

## 1. 文件用途

本文定義 GrooveScribe 什麼情況下可以稱為完整 V1。V1 不是「能跑通一次」；V1 必須能讓使用者在本機完成可重現、可理解、可恢復、可評估的完整工作流程。

## 2. V1 完整度總則

V1 必須同時滿足：

- 本機可啟動：使用者能依文件啟動 local web app。
- 本機可處理：AI runtime 在本機執行，不依賴雲端服務。
- 任務可追蹤：job 狀態、stage、progress、warnings、errors 可見。
- 結果可使用：MIDI、MusicXML、PDF 至少有明確成功 / 部分成功 / 失敗狀態。
- 失敗可診斷：使用者看到可理解訊息，工程師可從 log 找到 stage 與原因。
- 品質可評估：fixture tests 與人工評估可重現。
- 資料可管理：SQLite DB、artifacts、logs 有清楚位置、清理與備份策略。

## 3. 使用者體驗標準

### 啟動

- 提供一條主要啟動路徑。
- 啟動前或啟動後顯示 runtime preflight 結果。
- 缺少 ffmpeg、Demucs、ADTOF、MuseScore 或 Python package 時，顯示可操作的修復建議。

### 上傳

- 支援單首 MP3 / WAV。
- 前端與後端都檢查格式與大小。
- 上傳成功後導向 job result page。
- 上傳失敗時顯示原因，不暴露 stack trace。

### 處理中

- 顯示 queued / processing。
- 顯示目前 stage。
- 顯示 progress 或至少顯示 stage-level activity。
- 長時間無更新時，UI 顯示仍在處理或可能卡住的狀態，不直接誤判成功。

### 完成

- 顯示輸出 summary。
- 顯示 warnings 與準確度提醒。
- 顯示 MusicXML preview。
- 提供 MIDI、MusicXML、PDF 下載。
- PDF 失敗時，不影響 MIDI / MusicXML 使用。

### 失敗

- 顯示使用者可理解的錯誤訊息。
- 顯示可嘗試的下一步，例如換音檔、檢查 runtime、重試。
- 保留 internal error reference 或 pipeline log。
- 不暴露敏感本機路徑。

## 4. 工程標準

### Local-first runtime

- V1 預設 SQLite。
- V1 預設 local filesystem artifacts。
- V1 預設本機 job manager。
- Redis / Celery / PostgreSQL / S3 不得是 V1 必要條件。

### Job lifecycle

- job 建立、排隊、執行、完成、失敗有明確狀態。
- service restart 後，processing job 不應永久卡住；需標示 interrupted / failed 或提供 recovery。
- 每個 stage 有 started / completed / failed 記錄。
- pipeline log 可由 API 或本機工具讀取。

### Artifact contract

每個 completed 或 partially completed job 至少應能定位：

- original audio
- normalized wav
- drums stem
- raw MIDI
- processed MIDI
- drum events JSON
- MusicXML
- PDF，如果 export 成功
- pipeline log

Artifact metadata 至少包含：

- type
- storage key
- content type
- file size
- checksum 或等效 integrity metadata
- created time

### Error contract

錯誤至少包含：

- error code
- user-facing message
- retriable flag
- failed stage
- internal diagnostic reference

## 5. AI / 譜面品質標準

V1 不承諾出版級鼓譜，但必須達到可用草稿：

- 乾淨鼓軌 fixture 的主要 kick / snare / hi-hat groove 應可辨識。
- 授權真實鼓聲 fixture 應能產生可人工修正的 MIDI。
- 完整混音歌曲可產生有參考價值的主 groove，但允許漏判或誤判。
- hi-hat 開閉、ghost note、flam、ride / crash 區分可列為限制，不作 V1 完美標準。
- MusicXML 應可被 OpenSheetMusicDisplay 渲染。
- MusicXML 應可被 MuseScore 或相容工具開啟。
- PDF 應可下載並由一般 PDF viewer 開啟；renderer warning 需顯示。

## 6. 測試與驗收標準

V1 release 前應完成：

- Backend API tests。
- Local job manager tests。
- SQLite migration / schema tests。
- Storage adapter tests。
- Pipeline fixture tests。
- MusicXML / PDF generation tests。
- Frontend component / API mock tests。
- Browser smoke test：從 localhost UI 上傳短 fixture 並取得結果。
- Manual eval：至少一組 generated fixture 與一組授權真實鼓聲 fixture。
- Regression checklist：比較 pipeline version 與人工評估結果。

## 7. 文件標準

V1 文件必須包含：

- 本機安裝與啟動流程。
- Runtime preflight 說明。
- SQLite DB 位置與備份 / 重置方式。
- Artifacts 目錄結構與清理方式。
- 常見錯誤與排除。
- AI 準確度限制。
- Future optional capability 與 V1 主線的邊界。

## 8. 不足以稱為 V1 的狀態

以下情況只能稱為 POC 或 prototype，不應稱為完整 V1：

- 只能用 CLI 跑 pipeline，沒有完整 local web UI。
- 只能跑 mock pipeline，不能執行真 AI runtime。
- 只能產出 artifact，但 UI 無法查看 status / error / download。
- 失敗只留下 stack trace，沒有 user-facing error。
- 依賴 Redis / Celery / PostgreSQL 才能跑第一版。
- 沒有 fixtures、人工評估或 regression 記錄。
- 沒有 artifacts 清理、備份、重置說明。

## 9. Future Optional 不影響 V1

以下能力可以提高產品成熟度，但不是 V1 完整度必要條件：

- Tauri / Electron desktop shell。
- Cloud sync。
- Server deployment。
- User account / history sync。
- S3-compatible storage。
- Redis / Celery worker mode。
- PostgreSQL server mode。
- 多檔批次處理。
- 線上修譜。
