# GrooveScribe MVP 開發計畫

## 1. MVP 開發階段拆分

建議分成五個階段：

1. 本地 pipeline proof of concept。
2. 後端 API 與資料模型。
3. 前端上傳與結果頁。
4. 樂譜預覽與匯出。
5. 測試與部署。

每個階段都要產生可驗證成果，不要等到最後才整合 AI pipeline。

## 2. Phase 1：本地 Pipeline Proof of Concept

目標：在本機用 CLI / script 完成單首音檔從 MP3/WAV 到 MIDI / MusicXML / PDF 的流程。

工作項目：

- 建立 Python runtime。
- 安裝並驗證 `ffmpeg`。
- 安裝並驗證 Demucs。
- 安裝並驗證 ADTOF-pytorch。
- 準備 3 至 5 首測試音檔：乾淨鼓軌、一般歌曲、鼓聲不明顯歌曲。
- 實作本地 pipeline script：input audio path → output folder。
- 保留每個 stage artifact。
- 初版 MIDI post-processing：mapping、量化、輸出 processed MIDI。
- 初版 MusicXML / PDF 產生。

完成標準：

- 任一首 3 至 5 分鐘歌曲可完整跑完 pipeline。
- output folder 內包含 normalized WAV、drums stem、raw MIDI、processed MIDI、MusicXML、PDF。
- 工程師可用 README 指令重現。
- 至少一首乾淨鼓軌的 kick / snare / hi-hat 主 groove 可被辨識。

## 3. Phase 2：後端 API

目標：建立 API、資料模型、queue、worker，讓 pipeline 可透過 job 執行。

工作項目：

- 建立 FastAPI app。
- 建立 PostgreSQL schema 與 Alembic migration。
- 建立 storage adapter：LocalStorageAdapter。
- 建立 upload API。
- 建立 job status API。
- 建立 result API。
- 建立 download API。
- 建立 Celery + Redis worker。
- 將 Phase 1 pipeline 封裝成 worker task。
- 實作 job status / stage / progress 更新。
- 實作基本錯誤 response。

完成標準：

- 使用 API 上傳 MP3 / WAV 後可取得 job id。
- Worker 可消費 job 並執行完整 pipeline。
- 前端或 curl 可查詢 job status。
- completed job 可下載 MIDI / MusicXML / PDF。
- failed job 有明確 error code 與 error message。

## 4. Phase 3：前端上傳與結果頁

目標：建立可操作的 web MVP。

工作項目：

- 建立 React / Next.js app。
- 建立上傳頁。
- 加入檔案格式與大小提示。
- 呼叫 upload API。
- 建立 job result page。
- 使用輪詢查詢 job status。
- 顯示進度、目前 stage、錯誤狀態。
- completed 後顯示下載按鈕。

完成標準：

- 使用者可從瀏覽器上傳單首音檔。
- 上傳後自動進入 job result page。
- 分析中能看到狀態更新。
- 分析完成後能下載輸出檔案。
- 分析失敗時顯示可理解的錯誤訊息。

## 5. Phase 4：樂譜預覽與匯出

目標：讓使用者在網頁中看到簡單鼓譜，並穩定輸出 MusicXML / PDF。

工作項目：

- 決定 MusicXML generator：music21 或自訂 generator。
- 定義 drum notation mapping。
- 產生 percussion staff MusicXML。
- 整合 OpenSheetMusicDisplay。
- 確認 MusicXML 可被網頁預覽渲染。
- 整合 PDF export。
- 補上 PDF export 失敗時的 fallback 顯示。

完成標準：

- completed job 可在結果頁看到鼓譜預覽。
- MusicXML 可被 MuseScore 或相容軟體開啟。
- PDF 可下載並打開。
- 若 PDF 失敗，MIDI / MusicXML 仍可下載，job result 顯示部分成功狀態。

## 6. Phase 5：測試與部署

目標：讓 MVP 可被內部測試者穩定使用。

工作項目：

- 建立 unit tests。
- 建立 integration tests。
- 建立 pipeline fixture tests。
- 建立 Docker Compose。
- 建立 staging config。
- 設定 storage volume。
- 設定 worker timeout 與 concurrency。
- 設定 log 與基本 monitoring。
- 準備測試音檔清單與人工評估表。

完成標準：

- Docker Compose 可啟動 frontend、backend、worker、redis、postgres。
- 一首測試歌曲可從瀏覽器上傳並完成結果產出。
- 主要 API tests 通過。
- Pipeline 對固定測試音檔產出 artifacts。
- 工程師能根據 logs 追蹤失敗 stage。

## 7. 建議開發順序

```text
local pipeline
→ storage adapter
→ database models
→ queue + worker
→ upload / status API
→ download API
→ frontend upload
→ frontend status page
→ notation preview
→ PDF export
→ test suite
→ Docker deployment
```

## 8. 風險與優先處理

| 風險 | 優先處理方式 |
|---|---|
| Demucs / ADTOF 安裝困難 | Phase 1 先鎖定 runtime、版本與 Docker image |
| GPU 不穩或成本高 | 先支援 CPU proof of concept，但文件標示 GPU 建議 |
| MIDI 準確度不足 | 保留 raw MIDI、processed MIDI、人工評估表 |
| PDF export 不穩 | 讓 MusicXML 成為主要譜面輸出，PDF 可標記部分失敗 |
| 長任務卡住 | Worker timeout、stage progress、job failed 狀態 |
| storage 未來遷移 | 一開始就使用 storage adapter，不直接寫死 local path |
