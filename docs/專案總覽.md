# GrooveScribe 專案總覽

## 1. 專案目標

GrooveScribe 的 MVP 目標是建立一個「音訊轉鼓譜草稿」系統。使用者上傳單首 `WAV` 或 `MP3` 音檔後，系統在背景執行 AI 音訊處理 pipeline，產出：

- 基本鼓 MIDI 檔
- 簡單鼓譜 MusicXML 檔
- 簡單鼓譜 PDF 檔
- 網頁上的鼓譜預覽

第一版重點不是產生專業出版級鼓譜，而是快速產生可播放、可編輯、可下載的鼓譜初稿，降低使用者手動扒譜的時間成本。

## 2. MVP 定義

MVP 必須支援以下完整流程：

1. 使用者上傳單首 `MP3` 或 `WAV` 音檔。
2. API 建立 transcription job，立即回傳 job id，不在 request 期間執行長時間音訊處理。
3. Background worker 使用 `ffmpeg` 將音訊標準化為 pipeline 可處理的 WAV。
4. Worker 使用 Demucs 從完整歌曲中分離 `drums stem`。
5. Worker 使用 ADTOF-pytorch 將 `drums stem` 轉成 drum MIDI。
6. 系統對 MIDI 做基本後處理：tempo 對齊、量化、鼓件 mapping、去除明顯異常事件。
7. 系統產生可下載的 MIDI、MusicXML、PDF。
8. 前端結果頁顯示 job 狀態、處理進度、錯誤訊息與譜面預覽。
9. 使用者可以下載輸出檔案。

## 3. 非 MVP 範圍

第一版明確不做以下功能：

- 多人協作與共享編輯。
- 即時轉譜或串流音訊分析。
- 手機 App。
- 使用者線上付款、訂閱、點數制。
- 專業出版級排版。
- 人工智慧自動修譜到接近 100% 準確。
- 完整 DAW 編輯器。
- 鼓譜逐音符線上編輯器。
- 多樂器完整總譜轉寫。
- 使用者自訂 AI 模型訓練。
- 大量批次上傳。

## 4. 預期使用者

主要使用者：

- 鼓手：想快速取得歌曲鼓譜草稿，用於練習或改譜。
- 音樂老師：需要快速做教學用簡譜或 MIDI 範例。
- 編曲者 / 製作人：想把歌曲鼓組轉成 MIDI，放進 DAW 重新編輯或替換鼓音源。
- 音樂內容創作者：需要快速從歌曲產生節奏參考。

MVP 不以專業出版社、正式樂譜發行商為第一目標客群。

## 5. 使用情境

### 情境 A：鼓手練歌

鼓手上傳一首歌曲，等待數分鐘後取得簡單鼓譜與 MIDI。鼓手先用 PDF 看主 groove，再把 MIDI 匯入 DAW 或譜面軟體修正細節。

### 情境 B：音樂老師準備教材

老師上傳歌曲，系統輸出鼓譜草稿。老師下載 MusicXML 匯入 MuseScore 或其他譜面軟體，整理成適合學生程度的版本。

### 情境 C：製作人取得鼓 MIDI

製作人上傳 demo 或參考曲，系統產生基本 drum MIDI。製作人將 MIDI 匯入 Ableton、Logic、Cubase 等 DAW，替換鼓音源或修改節奏。

## 6. 第一版成功標準

第一版是否成功，以工程與產品兩個層面判斷。

### 工程成功標準

- 可在本地與 staging 環境完成一首音檔的完整 pipeline。
- 上傳 API 不被 Demucs / ADTOF-pytorch 長任務阻塞。
- 任務狀態可追蹤，包含 queued、processing、completed、failed。
- 每個 pipeline 階段都有明確輸入檔、輸出檔與錯誤紀錄。
- 系統可產生 MIDI、MusicXML、PDF 三種輸出。
- 儲存層可先用 local filesystem，並保留切換到 S3-compatible storage 的介面。

### 產品成功標準

- 針對乾淨鼓軌，能產生高度可參考的鼓 MIDI。
- 針對一般完整歌曲，能抓出主要 kick、snare、hi-hat groove。
- 使用者能理解輸出是「可用草稿」，不是保證完全準確的正式鼓譜。
- 使用者可以下載檔案並用第三方軟體繼續編輯。
- 一首 3 至 5 分鐘歌曲可在可接受時間內完成處理；MVP 目標為 GPU 環境下數分鐘內完成，CPU 環境允許更久但需顯示進度。
