# GrooveScribe 專案總覽

## 1. 專案目標

GrooveScribe V1 的目標是建立一個 production-ready local-first「音訊轉鼓譜」產品。使用者在自己的電腦上啟動服務，透過瀏覽器開啟 localhost 介面，上傳單首 `WAV` 或 `MP3` 音檔後，由本機 AI pipeline 產出：

- 可編輯的鼓 MIDI 檔
- 可交換與預覽的 MusicXML 檔
- 可下載與列印的 PDF 檔
- 網頁上的鼓譜預覽
- 可追蹤與可診斷的 pipeline log / artifacts

V1 的重點不是雲端部署，也不是只證明流程可跑通；重點是讓使用者能在本機穩定、可重現、可理解地完成完整工作流程。

## 2. 完整 V1 定義

完整 V1 必須支援以下流程：

1. 使用者在本機啟動 GrooveScribe service。
2. 使用者透過瀏覽器開啟 local web UI，例如 `http://localhost:5173`。
3. 使用者上傳單首 `MP3` 或 `WAV` 音檔。
4. Backend 建立本機 transcription job，立即回傳 job id，不在 request 期間執行長時間音訊處理。
5. 本機 job manager 管理 queued / processing / completed / failed 狀態。
6. Pipeline 使用 `ffmpeg` 將音訊標準化為可處理的 WAV。
7. Pipeline 使用 Demucs 從完整歌曲中分離 `drums stem`。
8. Pipeline 使用 ADTOF-pytorch 將 `drums stem` 轉成 drum MIDI。
9. 系統對 MIDI 做基本後處理：tempo 對齊、量化、鼓件 mapping、去除明顯異常事件。
10. 系統產生 MIDI、MusicXML、PDF 與 pipeline log。
11. 前端結果頁顯示 job 狀態、處理進度、錯誤訊息、警告與譜面預覽。
12. 使用者可以下載輸出檔案，並可在第三方 DAW 或譜面軟體中繼續修正。

## 3. V1 非目標

V1 明確不把以下能力列為主線：

- 雲端 Web App 或多人伺服器部署。
- SaaS 帳號、付款、訂閱、點數制。
- Cloud sync、分享連結、團隊協作。
- Tauri / Electron desktop shell。
- Redis / Celery 作為預設 queue。
- PostgreSQL 作為預設 database。
- S3-compatible storage 作為預設 artifacts storage。
- 手機 App。
- 即時轉譜或串流音訊分析。
- 完整 DAW 編輯器。
- 鼓譜逐音符線上編輯器。
- 專業出版級排版。
- 自動修譜到接近 100% 準確。
- 多樂器完整總譜轉寫。
- 大量批次上傳。

上述方向可作為 future / optional capability，但不可阻塞 V1 local-first 主線。

## 4. 預期使用者

主要使用者：

- 鼓手：想在本機快速取得歌曲鼓譜草稿，用於練習或改譜。
- 音樂老師：需要快速做教學用簡譜或 MIDI 範例。
- 編曲者 / 製作人：想把歌曲鼓組轉成 MIDI，放進 DAW 重新編輯或替換鼓音源。
- 音樂內容創作者：需要快速從歌曲產生節奏參考。

V1 不以專業出版社、正式樂譜發行商或多租戶 SaaS 客群為第一目標。

## 5. 使用情境

### 情境 A：鼓手練歌

鼓手在本機啟動 GrooveScribe，上傳一首歌曲，等待數分鐘後取得簡單鼓譜與 MIDI。鼓手先用瀏覽器看主 groove，再把 MIDI 或 MusicXML 匯入 DAW / MuseScore 修正細節。

### 情境 B：音樂老師準備教材

老師在自己的教學電腦上上傳歌曲，系統輸出鼓譜草稿。老師下載 MusicXML 匯入 MuseScore，整理成適合學生程度的版本。

### 情境 C：製作人取得鼓 MIDI

製作人在本機上傳 demo 或參考曲，系統產生 drum MIDI。製作人將 MIDI 匯入 Ableton、Logic、Cubase 等 DAW，替換鼓音源或修改節奏。

## 6. V1 成功標準

V1 是否成功，以產品完整度、工程穩定性與輸出品質一起判斷。

### 產品完整度

- 使用者可完成安裝、啟動、上傳、等待、預覽、下載、重新嘗試的完整流程。
- UI 能顯示 queued / processing / completed / failed。
- 失敗時有可理解錯誤，不暴露 stack trace 或本機敏感路徑。
- 使用者明確理解輸出是「AI 鼓譜草稿」，不是保證完全正確的正式鼓譜。

### 工程穩定性

- V1 預設使用 SQLite、本機 job manager、local filesystem artifacts。
- 長任務不阻塞 API request。
- 每個 pipeline stage 都有輸入、輸出、runtime、warning 與 error summary。
- 本機 service 重啟後，既有 jobs / artifacts 仍可查詢或被標示為可恢復 / 已失效。
- 固定 fixture tests、API tests、frontend smoke tests 與人工評估流程可重現。

### 輸出品質

- 針對乾淨鼓軌，能產生高度可參考的鼓 MIDI。
- 針對一般完整歌曲，能抓出主要 kick、snare、hi-hat groove。
- MusicXML 可被網頁預覽與 MuseScore 類工具開啟。
- PDF 可下載並打開；若 PDF renderer 失敗，MIDI / MusicXML 仍可保留並顯示部分成功狀態。

## 7. Future / Optional

V1 完成後可再評估：

- Tauri / Electron desktop shell。
- Cloud sync 與使用者歷史。
- Server deployment mode。
- Redis / Celery worker mode。
- PostgreSQL database mode。
- S3-compatible storage。
- 多模型比較與 fallback。
- 批次處理。
- 線上修譜與重新匯出。
