# GrooveScribe 產品需求文件

## 1. 核心功能需求

### 1.1 本機啟動

系統需提供 local-first 啟動能力。

需求：

- 使用者可在本機啟動 backend、frontend 與 job manager。
- 使用者透過瀏覽器開啟 localhost 介面。
- 啟動流程需檢查 AI runtime：ffmpeg、Demucs、ADTOF、MuseScore / PDF renderer。
- 若 runtime 不完整，UI 或命令列需顯示可操作的修復建議。

### 1.2 音檔上傳

系統需提供單首音檔上傳能力。

需求：

- 支援 `audio/mpeg`、`audio/wav`、`audio/x-wav`。
- 副檔名支援 `.mp3`、`.wav`。
- V1 每次只支援上傳一個音檔。
- 上傳完成後建立 transcription job。
- API 回傳 `job_id`，前端導向任務狀態頁。

### 1.3 本機背景分析任務

音訊分析必須由本機 job manager 執行，不在 API request lifecycle 中執行。

需求：

- API 只負責接收檔案、建立資料紀錄、派送本機 job。
- Job manager 依序執行 preprocessing、source separation、drum transcription、post-processing、notation generation、export。
- 每個階段要更新 job status、progress、current_stage。
- 任務失敗時需保留錯誤碼、錯誤摘要與 internal diagnostic reference。
- Service restart 後不應讓 job 永久卡在 processing。

### 1.4 結果頁

前端需提供結果頁。

需求：

- 顯示任務狀態。
- 顯示目前處理階段。
- 顯示 warnings 與準確度提醒。
- 完成後顯示輸出檔下載按鈕。
- 完成後顯示鼓譜預覽。
- 失敗時顯示可理解的錯誤訊息與下一步建議。

### 1.5 檔案下載

需求：

- 使用者可下載 MIDI。
- 使用者可下載 MusicXML。
- 使用者可下載 PDF。
- 下載 API 應驗證 job 是否 completed 或該 export 是否 available。
- 下載 API 不直接暴露內部檔案系統路徑。

## 2. 使用者流程

```text
啟動 GrooveScribe
→ 開啟 localhost UI
→ 選擇 MP3 / WAV
→ 前端呼叫 Upload API
→ API 寫入 SQLite 與 local storage
→ API 建立本機 job 並回傳 job_id
→ 前端導向 /jobs/{job_id}
→ 前端輪詢 Job Status API
→ 本機 job manager 執行 AI pipeline
→ job completed / partial success / failed
→ 前端顯示譜面預覽、warnings、下載按鈕或錯誤
→ 使用者下載 MIDI / MusicXML / PDF
```

## 3. 音檔上傳限制

V1 建議限制：

| 項目 | V1 限制 |
|---|---|
| 格式 | MP3、WAV |
| 單檔大小 | 100 MB 以內，實際限制可由設定調整 |
| 音檔長度 | 10 分鐘以內，實際限制可由設定調整 |
| 聲道 | mono / stereo 皆可，pipeline 統一處理 |
| 取樣率 | 任意輸入，pipeline 標準化 |
| 同時上傳 | 單檔 |
| 即時錄音 | 不支援 |

若音檔超出限制，API 應在建立任務前回傳 4xx 錯誤。

## 4. 輸出格式

### 4.1 MIDI

MIDI 是 V1 最重要的輸出，因為它可被 DAW 與譜面軟體後續編輯。

V1 MIDI 內容：

- 基本 drum events。
- General MIDI drum note mapping。
- 基本 velocity。
- 基本 tempo metadata。
- 不保證完整 humanization 或演奏手法。

### 4.2 MusicXML

MusicXML 用於樂譜交換與 web preview。

V1 MusicXML 內容：

- 4/4 為預設拍號；若 tempo / beat tracking 可取得可靠結果，再寫入更準確資訊。
- 使用 percussion staff。
- 將 kick、snare、hi-hat、tom、cymbal 映射到簡化鼓譜位置。
- 以可讀草稿為目標，不追求出版級 engraving。

### 4.3 PDF

PDF 用於直接查看與列印。

V1 PDF 內容：

- 由 MusicXML 轉出。
- 包含基本標題、tempo、鼓譜內容。
- 若轉換失敗，V1 允許 MIDI / MusicXML 成功但 PDF 標記 failed 或 completed_with_warning。

### 4.4 Web Preview

網頁預覽優先使用 MusicXML 渲染。

V1 預覽內容：

- 顯示單軌鼓譜。
- 顯示 MusicXML 載入狀態。
- MusicXML preview 失敗時保留下載按鈕與錯誤訊息。

## 5. 錯誤處理

### 5.1 上傳錯誤

常見錯誤：

- 檔案格式不支援。
- 檔案大小超過限制。
- 音檔 metadata 無法解析。
- 上傳中斷。
- local storage 寫入失敗。

回應應包含：

- `error.code`
- `error.message`
- `error.retriable`
- `error.details`，可選

### 5.2 Pipeline 錯誤

常見錯誤：

- `ffmpeg` 轉檔失敗。
- Demucs source separation 失敗。
- ADTOF-pytorch transcription 失敗。
- MIDI 後處理產生空結果。
- MusicXML / PDF 產生失敗。
- job timeout。
- runtime missing 或 executable 不可用。

Pipeline 錯誤需記錄：

- job id
- stage
- exception type
- summarized message
- input artifact key
- output artifact key, if any
- runtime version
- internal diagnostic reference

## 6. 使用者可預期的準確度說明

產品頁與結果頁需要清楚說明：

- 系統輸出的是「AI 鼓譜草稿」，不是保證完全正確的人工鼓譜。
- 乾淨鼓軌通常比完整混音歌曲準確。
- 混音中鼓聲被其他樂器遮蔽時，kick、snare、cymbal 可能漏判或誤判。
- hi-hat 開閉、ride / crash 區分、ghost note、flam、drag、roll 不保證準確。
- MIDI 和 MusicXML 建議使用者下載後再進行人工修正。

建議文案：

> GrooveScribe 會在你的電腦上產生可編輯的 AI 鼓譜草稿。主要 groove 通常可作為練習與改譜起點，但細節仍可能需要人工修正。

## 7. 資料與隱私

V1 預設：

- 音檔、MIDI、MusicXML、PDF 與 logs 留在使用者本機。
- 不需要帳號。
- 不上傳雲端。
- 使用者可清理 local artifacts。

Future optional cloud sync 或 server deployment 需另行設計資料保留、授權與刪除策略。

- server deployment。
- 使用者帳號與歷史紀錄。
- 專案分享連結。
