# GrooveScribe 測試策略

## 1. 測試目標

MVP 測試目標不是證明 AI 轉譜完美，而是確保：

- 系統流程穩定。
- API 與 worker 邊界清楚。
- 每個 pipeline stage 有可追蹤輸入與輸出。
- 成功任務可產生可下載檔案。
- 失敗任務有清楚錯誤狀態。
- 輸出的 MIDI / MusicXML / PDF 能被常見工具開啟。

## 2. 單元測試

建議測試範圍：

### 2.1 File Validation

- 接受 `.mp3`。
- 接受 `.wav`。
- 拒絕 `.exe`、`.txt`、未知 MIME type。
- 拒絕超過大小限制的檔案。

### 2.2 Storage Adapter

- put artifact。
- get artifact。
- generate download URL 或 download stream。
- local key mapping 正確。
- 不允許 path traversal。

### 2.3 Job State Transition

- uploaded → queued。
- queued → processing。
- processing → completed。
- processing → failed。
- failed job 必須有 error code。

### 2.4 MIDI Post-processing

- drum note mapping 正確。
- 同一鼓件極短時間重複事件會合併。
- 低 velocity noise 可依規則過濾。
- quantization 將事件對齊到預期 grid。
- empty MIDI 會回傳低信心或失敗狀態。

### 2.5 API Error Formatting

- 所有錯誤 response 都符合 `{ error: { code, message, retriable } }`。
- 不將 internal stack trace 回傳給前端。

## 3. Integration Test

Integration tests 應覆蓋 API、database、queue、storage 的整合。

測試案例：

1. 上傳合法音檔後建立 AudioFile 與 TranscriptionJob。
2. 上傳非法檔案回傳 400。
3. 上傳超大檔案回傳 413。
4. 查詢不存在 job 回傳 404。
5. job processing 時查詢 result 回傳 409。
6. completed job 可取得 exports metadata。
7. download API 對不存在 export 回傳 404。
8. worker 更新 progress 後 status API 能讀到。

## 4. Pipeline Test

Pipeline tests 使用固定 fixture 音檔驗證 artifacts 是否產生。

測試音檔類型：

- 乾淨鼓軌，短音檔，10 至 30 秒。
- 一般完整歌曲片段，30 至 60 秒。
- 無鼓或鼓聲很弱的音檔。
- 損壞音檔。
- 超短音檔。

驗證項目：

- `normalized.wav` 存在且可讀。
- `drums.wav` 存在且 duration 合理。
- `raw_drum.mid` 存在。
- `processed_drum.mid` 存在。
- `score.musicxml` 存在且 XML 可 parse。
- `score.pdf` 存在且可打開。
- job 最終狀態符合預期。

Pipeline tests 可分成：

- fast tests：mock Demucs / ADTOF，只測 orchestration。
- slow tests：實際跑 Demucs / ADTOF，用 nightly 或手動測試執行。

## 5. 不同音檔情境測試

### 5.1 乾淨鼓軌

預期：

- 主要 kick / snare / hi-hat 能被辨識。
- MIDI event count 合理。
- 譜面節奏接近原始 groove。

### 5.2 完整歌曲混音

預期：

- Demucs 能產生 drums stem。
- 主要 backbeat 可辨識。
- cymbal 與 hi-hat 可能有簡化或誤判。

### 5.3 鼓聲被蓋住的歌曲

預期：

- 系統仍能完成任務或回傳低信心結果。
- 結果頁顯示準確度提醒。

### 5.4 現場錄音

預期：

- 可能有 reverb 導致誤判。
- 不應造成 pipeline crash。

### 5.5 無鼓音檔

預期：

- 系統應回傳低信心或空結果警告。
- 不應產生看似完整但錯誤的鼓譜。

## 6. 準確度驗證方式

MVP 不需要建立大型 benchmark，但應建立小型人工標註資料集。

建議：

- 選 10 首 30 至 60 秒片段。
- 人工標註 kick、snare、hi-hat onset。
- 使用容忍時間窗，例如 ±50 ms。
- 計算 precision、recall、F1。
- 分類統計乾淨鼓軌與完整混音。

指標：

| 指標 | 說明 |
|---|---|
| Precision | 系統偵測出的鼓點有多少是真的 |
| Recall | 真實鼓點有多少被系統抓到 |
| F1 | Precision / Recall 綜合分數 |
| Timing Error | 偵測時間與人工標註時間差 |
| Usability Score | 人工評分：是否可作為修譜起點 |

## 7. 人工檢查流程

每次 release 前抽測固定音檔。

流程：

1. 上傳測試音檔。
2. 確認 job completed。
3. 播放原曲並打開 MIDI。
4. 檢查 kick / snare 主節奏是否合理。
5. 打開 MusicXML，確認小節與節奏沒有明顯崩壞。
6. 打開 PDF，確認可閱讀。
7. 填寫人工評估表。

人工評估表欄位：

- audio id
- genre
- clean drums or full mix
- kick score 1-5
- snare score 1-5
- hi-hat score 1-5
- notation readability 1-5
- overall usability 1-5
- notes

## 8. MVP 驗收標準

MVP 可驗收條件：

- API upload / status / result / download 功能完整。
- Worker 可成功處理至少 3 種測試音檔。
- completed job 產出 MIDI、MusicXML、PDF。
- failed job 有清楚錯誤碼與錯誤訊息。
- 前端可完成上傳、等待、預覽、下載。
- local filesystem storage 可運作，且 storage interface 可替換成 S3。
- 至少 5 首測試音檔完成人工檢查紀錄。
- 對乾淨鼓軌，主要 kick / snare pattern 應達到可用草稿程度。
- 對完整歌曲，結果頁需明確提示輸出可能需要人工修正。

## 9. Regression Test 建議

每次修改 pipeline 後，需固定比對：

- event count 是否大幅異常。
- processed MIDI 是否可 parse。
- MusicXML 是否可 parse。
- PDF 是否產生。
- 同一音檔的 job 是否能完成。

AI 輸出本身可能因模型或 runtime 版本不同而有差異，因此 regression test 應以可用性與 artifact 完整性為主，不應過度依賴 byte-by-byte snapshot。
