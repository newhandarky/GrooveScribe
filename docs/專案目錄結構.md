# GrooveScribe 建議專案目錄結構

```text
GrooveScribe/
├── frontend/
├── backend/
├── worker/
├── ai_pipeline/
├── docs/
├── storage/
├── tests/
├── scripts/
├── infra/
├── docker-compose.yml
└── README.md
```

## 1. frontend/

責任：

- Local Web App 使用者介面。
- 音檔上傳頁。
- Job status / result page。
- MusicXML 譜面預覽。
- MIDI / MusicXML / PDF 下載入口。

建議內容：

```text
frontend/
├── src/
│   ├── app/
│   ├── components/
│   ├── features/transcriptions/
│   ├── lib/api/
│   └── types/
└── package.json
```

## 2. backend/

責任：

- FastAPI API server。
- 上傳、狀態、結果、下載 API。
- SQLite 資料模型與 migrations。
- Storage interface。
- Local job manager enqueue interface。

建議內容：

```text
backend/
├── app/
│   ├── api/
│   ├── core/
│   ├── db/
│   ├── models/
│   ├── schemas/
│   ├── services/
│   └── storage/
├── migrations/
└── pyproject.toml
```

## 3. worker/

責任：

- 既有 worker / orchestration 實作。
- V1 主線應收斂為本機 job manager 或 pipeline orchestration module。
- Pipeline stage status updates。
- Retry / timeout / error handling。
- Celery / Redis 僅保留為 future server mode optional。

建議內容：

```text
worker/
├── app/
│   ├── celery_app.py
│   ├── tasks/
│   ├── pipeline_runner.py
│   └── error_mapping.py
└── pyproject.toml
```

## 4. ai_pipeline/

責任：

- 音訊處理與 AI 模型整合。
- ffmpeg preprocessing。
- Demucs adapter。
- ADTOF-pytorch adapter。
- MIDI post-processing。
- Notation generation。

建議內容：

```text
ai_pipeline/
├── preprocessing/
├── source_separation/
├── transcription/
├── midi/
├── notation/
├── adapters/
└── contracts/
```

重要原則：

- `ai_pipeline` 不應依賴 FastAPI request / response。
- `ai_pipeline` 可被 CLI、worker、tests 重複使用。
- 模型整合需透過 adapter，不能與 job domain 強耦合。

## 5. docs/

責任：

- 專案規劃文件。
- 架構文件。
- API 設計。
- 測試策略。
- 開發階段規劃。

目前文件：

```text
docs/
├── 專案總覽.md
├── 產品需求文件.md
├── 技術選型.md
├── 系統架構.md
├── 模組設計.md
├── AI音訊處理流程.md
├── API設計.md
├── 資料模型設計.md
├── 開發計畫.md
├── 測試策略.md
└── 專案目錄結構.md
```

## 6. storage/

責任：

- 本地開發用檔案儲存目錄。
- 保存 original audio、normalized audio、stems、MIDI、MusicXML、PDF、logs。

建議內容：

```text
storage/
├── local/
│   └── jobs/
└── README.md
```

注意：

- production 不應依賴 container ephemeral filesystem。
- 未來應切換到 S3-compatible storage。
- storage 內容通常不 commit 到 git。

## 7. tests/

責任：

- 單元測試。
- API integration test。
- Worker / queue test。
- Pipeline fixture test。
- 測試音檔與人工評估資料。

建議內容：

```text
tests/
├── unit/
├── integration/
├── pipeline/
├── fixtures/
│   ├── audio/
│   └── expected/
└── manual_eval/
```

## 8. scripts/

責任：

- 本地開發輔助指令。
- Pipeline proof of concept runner。
- 測試音檔處理。
- 清理過期 artifacts。

建議內容：

```text
scripts/
├── run_local_pipeline.py
├── inspect_midi.py
├── generate_musicxml.py
└── cleanup_storage.py
```

## 9. infra/

責任：

- Dockerfile。
- Docker Compose 設定。
- deployment config。
- GPU worker image 設定。

建議內容：

```text
infra/
├── docker/
│   ├── backend.Dockerfile
│   ├── worker.Dockerfile
│   └── frontend.Dockerfile
├── compose/
└── env/
```

## 10. 共享程式碼邊界建議

若 backend、worker、ai_pipeline 都是 Python，可考慮 monorepo package：

```text
packages/
└── groovescribe_common/
    ├── storage_contracts/
    ├── job_status/
    └── artifact_types/
```

但 V1 初期不一定需要過早拆 package。較務實做法是先保持 backend、worker、ai_pipeline 邊界清楚，等重複程式碼明顯增加後再抽共用套件。
