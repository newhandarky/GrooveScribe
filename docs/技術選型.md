# GrooveScribe 技術選型

## 1. V1 技術選型原則

GrooveScribe V1 是 production-ready local-first app。技術選型以本機可安裝、可重現、可診斷、可離線執行為優先。

V1 預設：

- Local Web App，而不是雲端 Web App。
- SQLite，而不是 PostgreSQL。
- 本機 job manager，而不是 Redis / Celery。
- Local filesystem artifacts，而不是 S3。
- 本機 Python AI runtime，而不是遠端 inference service。

Future optional 可以保留 server mode、desktop shell、cloud sync 與 SaaS 化，但不得成為 V1 主線依賴。

## 2. 前端語言與框架

建議：

- 語言：TypeScript
- 框架：React + Vite
- 狀態查詢：TanStack Query
- 檔案上傳：原生 `FormData` + fetch
- 譜面預覽：OpenSheetMusicDisplay，必要時搭配 VexFlow
- UI styling：可使用 Tailwind CSS、CSS Modules 或輕量 design system

理由：

- V1 是 localhost app，不需要 Next.js 的 server rendering 作為預設能力。
- Vite 啟動快、架構簡單，適合本機 app shell。
- TypeScript 能明確定義 API response、job status、export file type。
- TanStack Query 適合輪詢本機 job status。
- MusicXML 預覽優先用 OpenSheetMusicDisplay，因為 pipeline 輸出 MusicXML 後前端可直接渲染。

## 3. 後端語言與框架

建議：

- 語言：Python
- API 框架：FastAPI
- ORM / SQL：SQLAlchemy 2.x
- Migration：Alembic 或等效 SQLite migration runner
- Validation：Pydantic

理由：

- AI / 音訊處理工具鏈主要在 Python 生態。
- FastAPI 對檔案上傳、型別驗證、OpenAPI 文件支援成熟。
- Local Web App 仍需要穩定 API contract，方便前端與 pipeline 邊界清楚。
- Pipeline、runtime preflight、job manager 可以共用 Python domain models 與 storage adapter。

## 4. AI / 音訊處理使用的語言與套件

建議語言：Python。

核心套件：

| 用途 | 套件 / 工具 |
|---|---|
| 音訊轉檔 | ffmpeg |
| 音訊讀取 / 分析 | librosa、soundfile、torchaudio |
| 深度學習 | PyTorch |
| Source separation | Demucs |
| Drum transcription | ADTOF-pytorch |
| MIDI 讀寫 | pretty_midi、mido |
| MusicXML 產生 | music21 或自訂 MusicXML generator |
| PDF 產生 | MuseScore CLI 或 headless renderer |

## 5. 任務佇列與背景處理

V1 建議：

- Queue / job runner：本機 job manager。
- Concurrency：預設 1 個 pipeline job。
- Persistence：job 狀態寫入 SQLite。
- Recovery：啟動時檢查 stale `processing` job，標示 interrupted / failed 或提供重新執行。
- Timeout：由 local job manager 管理每個 stage 或整體 pipeline timeout。

理由：

- V1 是單機產品，不需要 Redis broker 來完成主要流程。
- 本機 job manager 降低安裝門檻與故障面。
- SQLite + filesystem 已足以保存本機 job 狀態與 artifacts。
- 長任務仍不應在 API request lifecycle 中執行。

Future optional：

- Redis / Celery：server mode、多 worker、GPU worker pool。
- RQ / Dramatiq：替代 queue implementation。
- Scheduler：清理過期 artifacts 或批次任務。

## 6. 資料庫與檔案儲存

### 6.1 資料庫

V1 預設：SQLite。

資料庫儲存：

- app settings metadata
- audio file metadata
- transcription job status
- stage timing
- export file metadata
- pipeline version / runtime version
- error summary

不直接儲存大型音訊或輸出檔案 blob。

SQLite 選型理由：

- 本機單使用者產品的安裝與備份成本最低。
- 不需要額外啟動 database service。
- 適合 local-first job history 與 metadata。
- 可透過 migration 管理 schema evolution。

Future optional：

- PostgreSQL：server mode、多使用者、hosted deployment。

### 6.2 檔案儲存

V1 預設：Local filesystem storage。

儲存內容：

- original upload
- normalized wav
- drums stem
- raw midi
- processed midi
- drum events json
- musicxml
- pdf
- logs / metadata json

程式碼不得在 UI 或 domain logic 中直接依賴任意 local path。API 與 pipeline 應透過 storage service 取得 read handle、write handle 或安全的 download stream。

Future optional：

- S3-compatible storage。
- cloud sync storage。
- external project folder export。

## 7. 樂譜顯示與輸出套件

建議：

- 後端產生 MusicXML：music21 或自訂 MusicXML generator。
- PDF 轉換：MuseScore CLI。
- 前端預覽：OpenSheetMusicDisplay。
- 若未來需要高度互動式自訂鼓譜編輯器，再引入 VexFlow。

選型原則：

- MusicXML 是交換格式與預覽格式的核心。
- PDF 是 MusicXML 的衍生輸出。
- 前端不要從 MIDI 直接推導譜面；V1 由 pipeline 產出單一 MusicXML 作為譜面真相來源。

## 8. 本機啟動與封裝

V1 目標：

- 提供一條本機啟動命令，啟動 backend、frontend dev/preview server 與 local job manager。
- 啟動前執行 runtime preflight，檢查 ffmpeg、Demucs、ADTOF、MuseScore、Python packages。
- 設定 app data 目錄，用於 SQLite DB、artifacts、logs。
- 文件化備份、清理與重置流程。

V1 不要求：

- staging deployment。
- cloud GPU worker。
- hosted frontend。
- reverse proxy。
- multi-user auth。

Future optional：

- Tauri / Electron desktop shell。
- Docker Compose server mode。
- hosted web deployment。

## 9. 為什麼選這些技術

- Local Web App：保留瀏覽器 UI 開發效率，同時不要求桌面 shell 或雲端部署。
- Python：最貼近音訊 AI 工具鏈，降低模型整合成本。
- FastAPI：適合本機 API boundary，型別清楚，能讓前端與 pipeline 解耦。
- SQLite：單機產品最小依賴，易備份、易除錯。
- 本機 job manager：滿足長任務非同步處理，同時避免 Redis / Celery 成為安裝門檻。
- Demucs：負責將完整歌曲分離出 drums stem，降低其他樂器對 drum transcription 的干擾。
- ADTOF-pytorch：聚焦 drum audio to MIDI，比多任務轉譜工具更適合作為 V1 主線。
- MusicXML：可被譜面軟體與網頁預覽工具消費，是 MIDI 到可讀鼓譜之間的重要交換格式。
- Storage Adapter：讓 V1 先穩定使用 local filesystem，未來不重寫業務邏輯即可改接 S3 或 cloud sync。
