# GrooveScribe 技術選型

## 1. 前端建議語言與框架

建議：

- 語言：TypeScript
- 框架：Next.js 或 React + Vite
- UI：Tailwind CSS 或 shadcn/ui
- 狀態查詢：TanStack Query
- 檔案上傳：原生 `FormData` + fetch / axios
- 譜面預覽：OpenSheetMusicDisplay，必要時搭配 VexFlow

理由：

- TypeScript 能明確定義 API response、job status、export file type。
- Next.js 適合產品化網站，React + Vite 適合較輕量的 app shell。
- TanStack Query 適合輪詢 job status。
- MusicXML 預覽優先用 OpenSheetMusicDisplay，因為後端輸出 MusicXML 後前端可直接渲染。

## 2. 後端建議語言與框架

建議：

- 語言：Python
- API 框架：FastAPI
- ORM：SQLAlchemy 2.x
- Migration：Alembic
- Validation：Pydantic

理由：

- AI / 音訊處理工具鏈主要在 Python 生態。
- FastAPI 對檔案上傳、型別驗證、OpenAPI 文件支援成熟。
- API 層與 worker 可共用 domain models、storage adapter、pipeline interface。

## 3. AI / 音訊處理使用的語言與套件

建議語言：Python。

核心套件：

| 用途 | 套件 / 工具 |
|---|---|
| 音訊轉檔 | ffmpeg |
| 音訊讀取 / 分析 | librosa、soundfile、torchaudio |
| 深度學習 | PyTorch |
| Source separation | Demucs |
| Drum transcription | ADTOF-pytorch |
| MIDI 讀寫 | pretty_midi、mido |
| MusicXML 產生 | music21 或自訂 MusicXML generator |
| PDF 產生 | MuseScore CLI 或 headless renderer |

## 4. 任務佇列與背景處理

MVP 建議：

- Queue：Redis
- Worker：Celery 或 RQ
- Scheduler：MVP 不需要；未來可加入清理過期檔案的 scheduled job

建議選擇：Celery + Redis。

理由：

- AI pipeline 任務時間長，不適合在 API request 中執行。
- Celery 支援 retry、timeout、task status、worker concurrency。
- Redis 作為 broker 對 MVP 足夠簡單。

任務處理原則：

- 每個 job 一個主要 pipeline task。
- task 內部切分 stage 並寫入 job progress。
- GPU worker concurrency 預設為 1，避免多任務搶 GPU memory。
- CPU worker 可分離執行轉檔、PDF 產生、清理檔案。

## 5. 資料庫與檔案儲存

### 5.1 資料庫

MVP 建議：PostgreSQL。

開發環境可用 SQLite，但 staging / production 應使用 PostgreSQL。

資料庫儲存：

- user metadata
- audio file metadata
- transcription job status
- stage timing
- export file metadata
- error summary

不直接儲存大型音訊或輸出檔案 blob。

### 5.2 檔案儲存

MVP 建議先實作 Storage Adapter：

- LocalStorageAdapter：本地 filesystem
- S3StorageAdapter：未來支援 S3-compatible storage

儲存內容：

- original upload
- normalized wav
- drums stem
- raw midi
- processed midi
- musicxml
- pdf
- logs / metadata json

程式碼不得在業務邏輯中直接依賴 local path。API 與 worker 應透過 storage service 取得 read handle、write handle 或 signed URL。

## 6. 樂譜顯示與輸出套件

建議：

- 後端產生 MusicXML：music21 或自訂 MusicXML generator。
- PDF 轉換：MuseScore CLI。
- 前端預覽：OpenSheetMusicDisplay。
- 若未來需要高度互動式自訂鼓譜編輯器，再引入 VexFlow。

選型原則：

- MusicXML 是交換格式與預覽格式的核心。
- PDF 是 MusicXML 的衍生輸出。
- 前端不要從 MIDI 直接推導譜面；MVP 由後端產出單一 MusicXML 作為譜面真相來源。

## 7. 部署方式

MVP 建議 Docker Compose：

- `frontend`：Next.js / React app
- `backend`：FastAPI API server
- `worker`：Celery worker，包含 AI runtime
- `redis`：queue broker
- `postgres`：database
- `storage`：local mounted volume；未來改 S3

Production 建議：

- API server 與 worker 分開部署。
- GPU worker 使用有 CUDA 的機器。
- CPU worker 與 GPU worker 分隊列。
- 檔案儲存切到 S3-compatible storage。
- 對長任務設定 timeout、重試上限、檔案保留時間。

## 8. 為什麼選這些技術

- Python：最貼近音訊 AI 工具鏈，降低模型整合成本。
- FastAPI：適合 API-first 架構，型別清楚，開發速度快。
- Celery + Redis：長任務處理成熟，能把 AI pipeline 從 request lifecycle 拆出來。
- Demucs：負責將完整歌曲分離出 drums stem，降低其他樂器對 drum transcription 的干擾。
- ADTOF-pytorch：聚焦 drum audio to MIDI，比多任務轉譜工具更適合作為第一版主線。
- MusicXML：可被譜面軟體與網頁預覽工具消費，是 MIDI 到可讀鼓譜之間的重要交換格式。
- Storage Adapter：讓 MVP 可先用 local filesystem，未來不重寫業務邏輯即可改接 S3。
