# 錯誤處理與使用者提示規格

> Level 2 功能細部設計。本文延伸 Level 1 產品與系統架構文件，不取代既有規劃。

## 功能目標

- 建立一致的 API error response、worker error mapping 與前端使用者提示。
- 避免將 stack trace、local path、模型內部錯誤直接暴露給使用者。
- 讓 failed job 可追蹤、可 debug、可給使用者下一步建議。

## 使用者情境

- 上傳格式錯誤時，使用者立即知道目前只支援 MP3 / WAV。
- AI pipeline 失敗時，使用者知道問題發生在音檔準備、鼓軌分離、鼓點辨識或譜面輸出。
- PDF 部分失敗時，使用者仍可下載 MIDI / MusicXML。

## 前端行為

- 所有 API 錯誤統一讀取 `error.message` 顯示。
- 依 `error.retriable` 決定是否顯示「重新上傳」或「稍後再試」。
- 對 known warnings 顯示非阻斷提示，例如「hi-hat 開閉不保證準確」。
- 不得顯示 raw exception、traceback、worker log。

## 後端行為

- 所有 API error response 使用 `{ error: { code, message, retriable, details } }`。
- API 將 validation error 映射為穩定錯誤碼。
- Download API 將 export missing / not ready 映射為 404 / 409。
- API 只回傳使用者可理解訊息，internal detail 存 log ref。

## Worker / Pipeline 行為

- Pipeline runner 包住每個 stage，捕捉 adapter exception 並轉成 domain error。
- 失敗時更新 `TranscriptionJob.status=failed`、`stage=failed` 或保留 failed stage、`error_code`、`error_message`、`error_stage`。
- 寫入 `internal_error_ref` 指向 pipeline log。
- 部分 export 失敗時，不一定讓整個 job failed；需依 export 是否為核心輸出判斷。

## 輸入資料

- API validation errors。
- Storage errors。
- Queue errors。
- Pipeline adapter errors。
- Export renderer errors。

## 輸出資料

- API error response。
- Job error fields。
- Pipeline log。
- Frontend toast / inline message。

## API 關聯

- `POST /api/v1/transcriptions`：上傳錯誤。
- `GET /api/v1/transcriptions/{job_id}/status`：failed job error。
- `GET /api/v1/transcriptions/{job_id}`：not completed / not found。
- `GET /api/v1/transcriptions/{job_id}/download/{type}`：export not ready / not found。

## Data Model 關聯

- `TranscriptionJob.error_code`。
- `TranscriptionJob.error_message`。
- `TranscriptionJob.error_stage`。
- `TranscriptionJob.internal_error_ref`。
- `ExportFile.error_code`。

## Storage / Artifact 關聯

- `jobs/{job_id}/logs/pipeline.json` 保存 internal error summary。
- Log 可含 adapter stderr 摘要，但不可直接回傳前端。
- 若 storage 本身不可用，至少要寫 database error fields。

## 狀態流轉

```text
validation error → no job created
queued/processing error → failed
export partial error → completed with ExportFile failed 或 failed，依核心輸出而定
```

## 錯誤情境

- 上傳：`INVALID_FILE_TYPE`、`FILE_TOO_LARGE`、`AUDIO_METADATA_UNREADABLE`。
- Storage：`STORAGE_WRITE_FAILED`、`STORAGE_READ_FAILED`、`ARTIFACT_NOT_FOUND`。
- Pipeline：`AUDIO_DECODE_FAILED`、`SOURCE_SEPARATION_FAILED`、`DRUM_TRANSCRIPTION_FAILED`、`MIDI_POST_PROCESSING_FAILED`、`NOTATION_GENERATION_FAILED`。
- Export：`PDF_EXPORT_FAILED`、`EXPORT_NOT_READY`、`EXPORT_NOT_FOUND`。

## 驗收標準

- 所有錯誤 response 符合統一格式。
- 前端不顯示 stack trace。
- Worker 失敗必定更新 job failed。
- 每個 failed job 都有 error code、message、stage。
- 已知 warnings 不阻斷下載。

## 測試案例

- API 測試：非法格式錯誤格式正確。
- API 測試：job not found。
- Worker 測試：Demucs adapter 拋錯映射為 `SOURCE_SEPARATION_FAILED`。
- Frontend 測試：failed status 顯示 message。
- 安全測試：API 不回傳 traceback 或 local path。
