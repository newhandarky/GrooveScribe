# 音檔上傳功能規格

> Level 2 功能細部設計。本文延伸 Level 1 產品與系統架構文件，不取代既有規劃。

## 功能目標

- 提供單首 MP3 / WAV 音檔上傳入口，建立 `AudioFile` 與 `TranscriptionJob`。
- 上傳 API 只做輕量驗證、檔案保存、資料紀錄與 queue enqueue，不執行 Demucs、ADTOF-pytorch 或任何長時間音訊分析。
- 成功後立即回傳 `202 Accepted`、`job_id`、`status_url`、`result_url`，讓前端導向結果頁。

## 使用者情境

- 鼓手在上傳頁選擇一首歌曲，輸入可選標題後送出。
- 使用者看到上傳進度，成功後自動進入 `/jobs/{job_id}`。
- 若檔案格式、大小或音檔 metadata 不合法，使用者在上傳頁直接看到可理解錯誤。

## 前端行為

- 提供 drag-and-drop 與檔案選擇按鈕，只允許單檔。
- 送出前檢查副檔名 `.mp3`、`.wav` 與大小不超過 100 MB；前端檢查僅作體驗優化，後端仍需重驗。
- 使用 `multipart/form-data` 呼叫 `POST /api/v1/transcriptions`，欄位為 `file` 與 optional `title`。
- 送出期間停用上傳按鈕，顯示上傳中狀態；收到 202 後導向結果頁。
- 收到 4xx / 5xx 時顯示 `error.message`，保留使用者選取檔案以便重新送出。

## 後端行為

- 驗證 HTTP content type、檔名副檔名、MIME type、檔案大小。
- 使用音訊 metadata inspector 做短 timeout 檢查，確認 duration 不超過 10 分鐘；若 metadata 無法解析，回傳 `422 AUDIO_METADATA_UNREADABLE`。
- 產生 `audio_file_id`、`job_id`，先將原始檔寫入 Storage Adapter，再建立資料庫紀錄。
- 建立 `TranscriptionJob` 時設定 `status=queued`、`stage=queued`、`progress=0`、`queued_at`。
- 將 `job_id` 與 pipeline config id 推入 queue；enqueue 失敗時不可留下不可追蹤的 orphan job，需標記 failed 或 rollback transaction。

## Worker / Pipeline 行為

- Upload API 不直接呼叫 worker 函式，只透過 Job Queue Module 派送任務。
- Worker 取得 `job_id` 後才讀取 original artifact 並開始 preprocessing。
- 若 worker 找不到 original artifact，job 進入 failed，錯誤碼為 `ORIGINAL_AUDIO_NOT_FOUND`。

## 輸入資料

- `file`: MP3 或 WAV，上限 100 MB，長度上限 10 分鐘。
- `title`: optional string，建議限制 120 字以內，只作顯示用途。
- 匿名 MVP 可不帶 user token；若未來有登入，從 auth context 寫入 `user_id`。

## 輸出資料

- 成功：`job_id`、`status=queued`、`status_url`、`result_url`、`created_at`。
- 失敗：統一錯誤格式 `{ error: { code, message, retriable, details } }`。
- 資料庫輸出：`AudioFile`、`TranscriptionJob`。
- Storage 輸出：`jobs/{job_id}/original/{filename}`。

## API 關聯

- `POST /api/v1/transcriptions` 是唯一上傳入口。
- 成功回傳 `202 Accepted`；格式錯誤回傳 400；檔案過大回傳 413；metadata 不可讀回傳 422。
- 此 API 不回傳分析結果，只回傳任務追蹤資訊。

## Data Model 關聯

- `AudioFile.original_filename`、`content_type`、`file_size_bytes`、`duration_seconds`、`original_storage_key`。
- `TranscriptionJob.audio_file_id`、`status`、`stage`、`progress`、`title`、`queued_at`。
- `User` 關聯在 MVP 可為 null。

## Storage / Artifact 關聯

- Artifact type：`original_audio`。
- Storage key：`jobs/{job_id}/original/{safe_filename}`。
- 檔名必須 sanitize，保留副檔名但移除 path separator 與控制字元。
- 業務邏輯只保存 storage key，不保存絕對 local path。

## 狀態流轉

```text
new request → validate upload → store original → create AudioFile → create TranscriptionJob(status=queued, stage=queued) → enqueue job → return 202
```

## 錯誤情境

- `INVALID_FILE_TYPE`: 副檔名或 MIME type 不支援。
- `FILE_TOO_LARGE`: 超過 100 MB。
- `AUDIO_TOO_LONG`: metadata 顯示超過 10 分鐘。
- `AUDIO_METADATA_UNREADABLE`: 無法解析音訊 metadata。
- `STORAGE_WRITE_FAILED`: 原始檔寫入失敗。
- `QUEUE_ENQUEUE_FAILED`: 任務無法推入 queue。

## 驗收標準

- 合法 MP3 / WAV 可成功建立 job 並回傳 202。
- API response 不等待 AI pipeline 完成。
- 資料庫有對應 `AudioFile` 與 `TranscriptionJob`。
- Storage 中存在 original artifact。
- 非法格式、超大檔案、不可讀音檔皆有穩定錯誤 response。

## 測試案例

- 單元測試：檔名、MIME type、size validator。
- 整合測試：上傳合法 MP3 建立 job 並 enqueue。
- 整合測試：上傳 `.txt` 回傳 `INVALID_FILE_TYPE`。
- 整合測試：Storage adapter 寫入失敗時回傳 `STORAGE_WRITE_FAILED`。
- 安全測試：檔名含 `../` 不可造成 path traversal。

## 非 MVP 範圍

- 批次上傳。
- 直接上傳到 S3 signed URL。
- 即時錄音上傳。
- 上傳前瀏覽器端音訊波形分析。
